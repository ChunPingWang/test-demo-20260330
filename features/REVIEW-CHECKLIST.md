# 審查檢查清單 (Review Checklist)

Customer Management API - Gherkins BDD 測試案例專案

---

## 📋 API 完整性檢查

| 項目 | 檢查點 | ✅/❌ | 備註 |
|-----|--------|------|------|
| **端點覆蓋** | 是否覆蓋所有 API 端點？ | ☐ | 檢查API清單中的5個端點 |
| **HTTP 方法** | HTTP 方法是否正確？ | ☐ | GET, POST, PUT, DELETE |
| **路徑參數** | 路徑參數是否正確標記？ | ☐ | {id} 參數 |
| **功能描述** | 每個 API 的功能描述是否清晰？ | ☐ | 中文或英文已定義 |
| **Base URL** | Base URL 是否支援環境變數參數化？ | ☐ | 應可由 BASE_URL 環境變數配置 |

---

## 🧪 測試覆蓋範圍檢查

### 正向測試 (Happy Path)
| 項目 | 檢查點 | ✅/❌ | 備註 |
|-----|--------|------|------|
| **GET /api/customers** | 成功取得列表？ | ☐ | 正常且空列表 |
| **GET /api/customers/{id}** | 成功取得单筆記錄？ | ☐ | 含時間戳記驗證 |
| **POST /api/customers** | 建立新客戶成功？ | ☐ | 全部欄位、最小欄位、部分欄位 |
| **PUT /api/customers/{id}** | 更新客戶成功？ | ☐ | 全部更新、部分更新、清空欄位 |
| **DELETE /api/customers/{id}** | 刪除客戶成功？ | ☐ | 驗證刪除後查詢失敗 |

### 負向測試 (Error Cases)
| 項目 | 檢查點 | ✅/❌ | 備註 |
|-----|--------|------|------|
| **缺少必填欄位** | 是否測試所有必填欄位的缺失？ | ☐ | name, email |
| **格式驗證** | 是否測試無效的格式？ | ☐ | Email 格式、ID 格式 |
| **唯一性約束** | 是否測試重複的唯一值？ | ☐ | Email 重複檢查 |
| **資源不存在** | 是否測試 404 情況？ | ☐ | 取得/更新/刪除不存在的資源 |
| **400 Bad Request** | 是否測試參數驗證失敗？ | ☐ | 格式、長度、類型錯誤 |

### 邊界值測試 (Boundary Cases)
| 項目 | 檢查點 | ✅/❌ | 備註 |
|-----|--------|------|------|
| **最小值** | 是否測試字段最小長度？ | ☐ | name: 1 字、ID: 1 |
| **最大值** | 是否測試字段最大長度？ | ☐ | name: 50、email: 100 |
| **超出範圍** | 是否測試超出範圍値？ | ☐ | name: 51、email: 101 |
| **特殊字符** | 是否測試特殊字符？ | ☐ | ', -, +, . |
| **空值** | 是否測試空值和 null？ | ☐ | 必填 vs 選用欄位 |

## 📝 Gherkins 品質檢查

| 項目 | 檢查點 | ✅/❌ | 備註 |
|-----|--------|------|------|
| **語法正確性** | Gherkins 語法是否正確？ | ☐ | Feature, Scenario, Given-When-Then |
| **Scenario 標題** | 標題是否清晰且描述性強？ | ☐ | 避免 "TC001"、"Test" 等 |
| **Given-When-Then 結構** | 是否遵循 GWT 結構？ | ☐ | 前置條件、操作、預期結果 |
| **業務語言** | 是否使用業務語言而非技術細節？ | ☐ | "驗證 email 格式有效" vs "Regex match" |
| **Step 重用** | 是否存在重複的 Step？ | ☐ | 考慮使用 Scenario Outline |
| **前置條件** | 是否適當使用 Background？ | ☐ | 共同的 Given 應在 Background |
| **資料格式** | 是否使用清晰的資料表格？ | ☐ | Markdown 表格或 pipe-separated |

## 🔒 認證與授權檢查

| 項目 | 檢查點 | ✅/❌ | 備註 |

## 📊 數據模型檢查

| 項目 | 檢查點 | ✅/❌ | 備註 |
|-----|--------|------|------|
| **Customer 欄位** | 所有欄位是否都測試過？ | ☐ | id, name, email, phone, address, createdAt, updatedAt |
| **欄位驗證** | 是否測試每個欄位的驗證規則？ | ☐ | 類型、長度、格式 |
| **時間戳記** | 是否驗證 createdAt/updatedAt？ | ☐ | ISO 8601 格式 |
| **ID 生成** | 是否驗證 ID 自動生成？ | ☐ | POST 201 回應包含 ID |

## 🎯 場景執行順序建議

| 優先級 | Scenario | Feature 檔 | 理由 |
|--------|----------|-----------|------|
| 1️⃣ P0 | 所有 happy path | 所有 | 驗證基礎功能 |
| 2️⃣ P1 | 必填欄位驗證 | create, update | 確保約束強制執行 |
| 3️⃣ P2 | 404 Not Found | get, delete | 驗證錯誤狀態碼 |
| 4️⃣ P3 | Email 驗證 | create, update | 確保格式檢查 |
| 5️⃣ P4 | 邊界值 | create, update, get | 驗證長度限制 |
| 6️⃣ P5 | 特殊字符 | create, update | 驗證編碼處理 |
| 7️⃣ P6 | 冪等性 | delete | 驗證幂等性 |

## ✅ 質量指標

### 覆蓋率
- [ ] **行覆蓋率**: 所有 API 端點都被測試
- [ ] **分支覆蓋率**: 所有錯誤路徑都被測試
- [ ] **場景覆蓋率**: Positive + Negative + Boundary

### Gherkins 品質
- [ ] **可讀性**: 任何人都能理解 Scenario
- [ ] **可維護性**: Scenario 獨立、易於更新
- [ ] **可執行性**: 所有 Scenario 都能被自動化

### 文檔品質
- [ ] **完整性**: 所有 API 都有文檔
- [ ] **準確性**: 文檔與實際 API 一致
- [ ] **清晰性**: 描述清楚、易於理解
