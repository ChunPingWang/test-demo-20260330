## ❌ 失敗的測試詳情 (2)

### 1. POST /api/customers - 13-驗證 name 欄位長度限制 - A51.bru

**基本資訊**：
- 端點：POST /api/customers
- 優先級：Low
- 響應時間：245 ms
- 執行時間：2026-03-30 15:34:12

**失敗詳情**：
```
預期狀態碼：400 Bad Request
實際狀態碼：201 Created

失敗原因：
後端未正確驗證 name 欄位的長度限制。根據 API 文件，
name 欄位最大長度為 50 個字符，但後端接受了 51 個字符的
輸入並成功建立了客戶。

測試請求：
{
  "name": "A" 重複 51 次,
  "email": "test@example.com"
}

預期結果：400 Bad Request
實際結果：201 Created

響應體：
{
  "id": "cust_88888",
  "name": "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA...",
  "email": "test@example.com",
  "createdAt": "2026-03-30T15:34:12Z",
  "updatedAt": "2026-03-30T15:34:12Z"
}
```

**建議**：
- 檢查後端 API 中 Customer entity 的 name 欄位驗證規則
- 確認資料庫欄位定義是否與 API 文件一致（VARCHAR(50)）
- 建議修復：在請求驗證層添加 @Size(max = 50) 或類似的驗證

**相關測試**：
- 12-驗證 name 欄位長度限制 - A50.bru ✅ (通過)
- 14-驗證 name 欄位長度限制 - empty.bru ✅ (通過)

### 2. PUT /api/customers/{id} - 05-更新失敗 - 無效的 Email 格式.bru

**基本資訊**：
- 端點：PUT /api/customers/{id}
- 優先級：Medium
- 響應時間：580 ms
- 執行時間：2026-03-30 15:39:05

**失敗詳情**：
```
預期狀態碼：400 Bad Request
實際狀態碼：200 OK

失敗原因：
後端在更新客戶時未驗證 email 欄位格式。無效的 email
格式被接受並保存到資料庫。

測試請求：
PUT /api/customers/cust_12345
{
  "name": "John Doe",
  "email": "invalid.email.format"
}

預期結果：400 Bad Request
實際結果：200 OK

響應體：
{
  "id": "cust_12345",
  "name": "John Doe",
  "email": "invalid.email.format",
  "updatedAt": "2026-03-30T15:39:05Z"
}
```

**根本原因分析**：
- Email 驗證邏輯可能只在 POST (建立) 操作中實現
- PUT (更新) 操作缺少相同的電子郵件格式驗證
- 後端沒有統一的驗證機制

**建議**：
- 在 UserValidator 或共同驗證器中實現電子郵件驗證
- 確保 POST 和 PUT 操作都適用相同的驗證規則
- 考慮使用 Spring 的 @Email annotation 或自訂正則表達式驗證

**相關測試**：
- 07-建立客戶失敗 - 無效的 Email 格式.bru ✅ (在 POST 中通過)
- 04-更新失敗 - 缺少 name 欄位.bru ✅ (其他驗證通過)
